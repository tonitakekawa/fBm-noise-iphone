# 目論見：音と画の同期 fBm（modality 横断ノイズ）

## 概要

すでに実装した fBm 音響ノイズに加えて、**同じ fBm 構造で駆動される動的な砂嵐画面**を追加する。
音と画は**共通の乱数源（オクターブ群）から駆動**され、低周波オクターブの揺らぎが両 modality で同期する。

つまり、画面が大きくうねった瞬間、耳に届く音も大きくうねる。
人間は modality を超えた構造の同期を、頭で考えるより先に身体で感じる。

## 設計の核：共有オクターブソース

通常、音の fBm と画の fBm は別々に実装するが、それでは独立した二つのノイズになってしまう。
ここでは **一つの fBm オクターブ群を中央に置き、音と画がそれぞれサンプリング**する構造にする。

```
              ┌──────────────────────┐
              │  SharedFBmSource     │
              │  ┌────────────────┐  │
              │  │ Octave 0 (高)   │  │
              │  │ Octave 1       │  │
              │  │ Octave 2       │  │
              │  │ ...            │  │
              │  │ Octave 7 (低)   │  │
              │  └────────────────┘  │
              └─────────┬────────────┘
                        │
            ┌───────────┴───────────┐
            ▼                       ▼
      ┌───────────┐           ┌───────────┐
      │  Audio    │           │  Visual   │
      │  Sampler  │           │  Sampler  │
      └───────────┘           └───────────┘
            │                       │
       AVAudioEngine          Metal Shader
       (44100Hz)              (60fps)
```

**ポイント**：
- 低周波オクターブ（更新が遅い）は、音と画の両方に**同じ大きな揺らぎ**を与える
- 高周波オクターブは、それぞれの modality 固有の細部に使う
- 同期は「強制」ではなく「構造的に同じ」になる

## アーキテクチャ詳細

### 1. SharedFBmSource

中央の fBm 状態管理クラス。`Actor` または `@MainActor` で thread-safe に。

- 8〜10 オクターブの乱数値を保持
- 各オクターブが「次に更新されるべき時刻」を持つ
- `value(at: TimeInterval) -> Float` で任意時刻のサンプル値を返す
- 内部的に時刻が進むと、必要なオクターブを更新する

#### 重要：時間ベースの更新

これまでの音版 Voss-McCartney は「サンプルカウンタの最下位ビット」で更新オクターブを決めていた。
今回は音と画で**サンプリングレートが違う**（音 44100Hz、画 60Hz）ので、サンプルカウンタではなく **絶対時刻** で更新を制御する。

各オクターブ k は、`baseInterval * 2^k` 秒ごとに更新する。
- `baseInterval = 1/44100` 秒（音の最小単位）
- オクターブ 7 なら 128/44100 ≈ 2.9ms ごとに更新
- オクターブ 15 なら ≈ 0.74 秒ごとに更新

実装では、各オクターブが「次の更新時刻」を持っていて、サンプリング要求が来たら必要な分だけ更新を進める。

### 2. AudioSampler（既存の FBmGenerator を改造）

`AVAudioSourceNode` のレンダーブロック内で、`SharedFBmSource` から現在時刻のサンプルを取得して波形に変換。

- レンダーブロックには現在時刻が `AudioTimeStamp` で渡される
- これを `SharedFBmSource` の時刻系に変換して `value(at:)` を呼ぶ
- ステレオ用に左右で**わずかにオフセット**した時刻を使うと、空間的広がりが自然になる（例：右チャンネルは +0.3ms）

### 3. VisualSampler（Metal シェーダ）

Metal シェーダでフルスクリーンの fBm 砂嵐を描画。

- 毎フレーム（60fps）、`SharedFBmSource` から現在時刻のオクターブ値群を取得
- それを Metal の uniform バッファに転送
- フラグメントシェーダで、各ピクセルの (x, y) 座標と uniform から fBm 値を計算
- グレースケールで描画

#### シェーダ内の fBm 計算

ここが一番面白いところ。**画面の空間 fBm を、時間方向に動かす**ために：

```metal
fragment float4 fbmShader(
    VertexOut in [[stage_in]],
    constant FBmUniforms& uniforms [[buffer(0)]]
) {
    float2 uv = in.position.xy / uniforms.resolution;
    float total = 0.0;
    float amplitude = 1.0;
    float frequency = 1.0;

    // 8 オクターブの空間 fBm
    for (int i = 0; i < 8; i++) {
        // 各オクターブに「時間方向のオフセット」として
        // SharedFBmSource の対応するオクターブ値を加える
        float2 p = uv * frequency + uniforms.timeOffsets[i];
        total += hash2D(p) * amplitude;
        amplitude *= 0.5;
        frequency *= 2.0;
    }
    return float4(total, total, total, 1.0);
}
```

**`timeOffsets[i]` こそが共有点**。
これは `SharedFBmSource` のオクターブ i の値を渡すもので、CPU 側で時間とともに揺らぐ。
低周波オクターブが揺らぐと画面全体の低周波構造がスライドし、それが**音の低周波オクターブの揺らぎと同期**する。

### 4. ContentView の変更

- 既存のホワイトノイズボタン
- 既存の fBm（音のみ）ボタン
- **新規：「同期 fBm」ボタン**（押すと音 + 画面砂嵐が同時に始まる）
- 同期 fBm 中は全画面が砂嵐表示になる（既存 UI は半透明オーバーレイ等で残す）

## 技術的な要所

### a. Metal の使い方

iOS / macOS Multiplatform で Metal を使う場合：

- `MTKView` を `UIViewRepresentable` / `NSViewRepresentable` でラップして SwiftUI に組み込む
- もしくは SwiftUI の `MetalView`（iOS 17+ / macOS 14+）を使う
- フラグメントシェーダだけのシンプルな構成で十分（頂点シェーダはフルスクリーン三角形）

Multiplatform で動かすには、`#if canImport(UIKit)` / `#if canImport(AppKit)` の分岐が必要だが、Metal 自体は両 OS で同じ API。

### b. スレッド安全性

`SharedFBmSource` は：
- オーディオスレッド（リアルタイム、優先度高）から呼ばれる
- メインスレッド（描画）からも呼ばれる

ここで `Actor` を使うとオーディオスレッドが await でブロックされうるので NG。
**lock-free な実装**にする：

- 各オクターブの値を `atomic` または `os_unfair_lock` で保護
- もしくは、書き込みは別スレッドで行い、読み込みはコピーされたスナップショットを使う
- 推奨：オーディオスレッドが値を更新し、描画スレッドは読み取りのみ（read-mostly）

### c. 時刻の同期

- `mach_absolute_time()` または `CACurrentMediaTime()` を共通の時計に使う
- 音側は `AVAudioTime` から `hostTime` 経由で同じ時計に変換可能
- 画側は `CADisplayLink` の `targetTimestamp` を使う

### d. パフォーマンス

- 8 オクターブの空間 fBm をフラグメントシェーダで計算 → 4K 画面でも 60fps 余裕
- CPU 側の `SharedFBmSource` は毎フレーム 1 回しか呼ばれないので軽量
- オーディオスレッドは 44100Hz で値取得するが、ロックフリーなら問題なし

## 動作確認のポイント

1. **同期感**：画面の大きな揺らぎと音の低周波うねりが、同じタイミングで起きるか
2. **画の見た目**：アナログテレビの砂嵐っぽいか（残光感が欲しければ前フレームを少し残す混合を入れる）
3. **CPU/GPU 負荷**：寝室で一晩点けっぱなしにしても発熱しないか
4. **音と画のレイテンシ差**：人間の知覚的同時性窓（±100ms）に収まっているか
5. **長時間再生での破綻**：時刻のオーバーフロー、float の精度劣化に注意

## 任意の追加（実装後の遊び）

- **残光**：前フレームの 0.7 倍を新フレームに加算 → ブラウン管の蛍光体っぽさ
- **走査線**：水平方向に薄い縞を入れる → アナログ TV っぽさ強化
- **色付け**：完全グレースケールではなく、ごくわずかに青寄り or 緑寄り → ノスタルジー
- **自動フェード**：開始時/停止時に音量 + 画面輝度をクロスフェード

ただし、過剰演出は「composed viewer 前提のフィルタ」になりかねないので、最小限がよい。

## 作業順序

1. `SharedFBmSource.swift` を実装（時刻ベースのオクターブ管理）
2. 既存の `FBmNoisePlayer` を `SharedFBmSource` から駆動するよう改修（このとき音のみで動作確認）
3. `fbm.metal` フラグメントシェーダを書く（CPU 入力なしの静止 fBm をまず確認）
4. `FBmMetalView.swift` で `MTKView` を SwiftUI に組み込む
5. `SharedFBmSource` の値を Metal の uniform に渡す経路を作る
6. `SyncedFBmPlayer.swift` で音と画を統合
7. `ContentView.swift` にボタンと全画面表示を追加
8. 同期感の調整（オフセット、スケール、感度）

## 哲学的メモ

これは「視聴覚演出」ではなく **「構造の二重提示」** である。
同じ fBm 構造が、聴覚系を通れば音になり、視覚系を通れば砂嵐になる。
両者を同時に提示することで、観察者は「**どちらでもない、その背後にある構造**」を、合成的に推論せざるを得なくなる。

これは Marr の三段階（計算理論 / 表現とアルゴリズム / 実装）における「計算理論」のレベルが、二つの「実装」を通して逆に立ち現れる装置とも言える。

ただし、聴き手・見手にこれを言葉で説明する必要はない。
ただ提示するだけでよい。気づくかどうかは身体に委ねる。
